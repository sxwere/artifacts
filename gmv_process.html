01 问题定义
要回答的问题

某 D2C 品牌 2025 年 11 月至 12 月 GMV 出现环比下滑。作为分析师，需要回答：下滑的量有多少，是什么导致的，是流量问题、转化问题，还是客单价问题？

光知道"GMV 掉了"对业务没有价值。需要把下滑量分解到可操作的因子上——产品团队能接着看转化率，运营团队能接着看渠道，这套分析才有用。

数据情况

原始数据是用户行为日志，每一行是一个用户在某个时间点的一次访问记录，包含渠道、活动类型、漏斗各阶段状态（是否访问首页、是否查看商品、是否加购、是否结算、是否完成支付）以及订单金额。

同一个用户可能在同月内有多条记录。这个特性后续在计算 UV 时需要特别处理。

02 连锁替代法拆解三因子
为什么用连锁替代法

GMV = UV × CVR × AOV，三个因子在两个月之间都发生了变化，我需要量化每个因子单独贡献了多少下滑。

最直觉的做法是算每个因子自己的变化率，但这忽略了因子之间的交叉项（即因子 A 变化的同时 B 也在变，两者的乘积部分归给谁？）。连锁替代法的做法是：每次只替换一个因子，其余保持基期值不变，通过前后差额剥离出每个因子的纯贡献，交叉项被顺序分摊进去。

遇到的问题 · UV 算成了行为点击数

最初我在计算 UV 时，直接使用了 count() 来统计行数。但原始数据中同一用户在同一个月可能有多行记录（不同渠道进来、不同日期访问），这导致 UV 实际统计的是行为点击次数而非用户人数，严重虚高。

修正方法是改用 nunique() 对 user_id 做去重计数，并且在计算各漏斗步骤的 UV 时，先 groupby user_id 取各步骤的 max（即只要该用户这个月有过一次该行为就算完成），再 sum 得到人头数。

分析结论

UV 流量下滑贡献了 GMV 降幅的 123%，即使 CVR 和 AOV 都轻微正向，也完全无法对冲流量的流失。问题的根本在流量侧，而非转化侧或客单价侧。

chain_substitution.py · 连锁替代法核心逻辑 
# 提取 11 月（基期）和 12 月（目标期）的整行指标数据
nov = monthly_analysis[monthly_analysis['month'] == '2025-11'].iloc[0]
dec = monthly_analysis[monthly_analysis['month'] == '2025-12'].iloc[0]

# GMV 实际总变动量（分母）
total_gmv_change = dec['gmv'] - nov['gmv']

# 连锁替代法：每次只换一个因子，其余保持基期值
# 因子 A（UV）：UV 变，CVR/AOV 用 11 月值
uv_effect  = (dec['uv'] - nov['uv']) * nov['crv'] * nov['aov']

# 因子 B（CVR）：UV 已用 12 月值，CVR 变，AOV 仍用 11 月值
crv_effect = dec['uv'] * (dec['crv'] - nov['crv']) * nov['aov']

# 因子 C（AOV）：UV/CVR 都用 12 月值，只有 AOV 在变
aov_effect = dec['uv'] * dec['crv'] * (dec['aov'] - nov['aov'])

# 各因子贡献率（三者相加应等于 100%）
uv_contribution  = uv_effect  / total_gmv_change
crv_contribution = crv_effect / total_gmv_change
aov_contribution = aov_effect / total_gmv_change
03 UV 深挖：用户类型拆解
为什么要拆用户类型

UV 整体下滑，但"哪类用户少了"决定后续要采取什么动作。新客流失和留存客户流失的对策完全不同：前者看获客渠道，后者看复购和召回。

我把每个月的活跃用户分成三类：新客户（历史从未出现过）、留存客户（上个月也来了）、唤醒客户（上上个月或更早来过，上个月没来）。

遇到的问题 · 数据穿越（Look-ahead bias）

最初的写法是直接用全量数据判断某用户"是否曾经来过"，这在当前场景看上去没问题，但如果换一个没有按时间排好的写法，或者后续扩展到滚动窗口时，很容易用到"未来"数据给用户打标签，把本来是新客户的人错误判断为老客户。

我的解决方案是在函数内部先对 df 切片，只保留 target_month 及更早的记录，再在这个切片内判断用户身份——从根本上切断了未来数据的干扰，函数逻辑可以复用到任意月份而不出错。

遇到的问题 · 渠道归因下的新客人数膨胀

在做"新客户 × 渠道"交叉分析时，我发现渠道维度加总的新客人数比 UV 层的新客人数多。排查后发现：同一个新客用户当月可能从多个渠道进来，在渠道维度就会被计入多次。

这个现象本身是合理的——用户可以多渠道触达。但要注意的是：渠道维度的数字用于分析各渠道的相对贡献，而不能用于验证新客总人数。我在报告中说明了这一点，避免后续读者产生疑惑。

分析结论

12 月 UV 下滑中，新客户流失是主要来源。留存客户和唤醒客户数量变化相对有限。这把问题锁定在获客侧，而不是存量用户的留存能力。

user_tagging.py · 防数据穿越的用户分类 
def get_monthly_customer_tags(df, target_month, last_month):
    """严格隔离未来数据，只基于历史判定用户身份"""
    # ① 切断未来数据
    history_df = df[df["month"] <= target_month]

    target_users     = set(df[df["month"] == target_month]["user_id"])
    last_month_users  = set(df[df["month"] == last_month]["user_id"])
    before_last_users = set(df[df["month"] < last_month]["user_id"])

    res_list = []
    for uid in target_users:
        if uid not in last_month_users and uid not in before_last_users:
            tag = "新客户"
        elif uid in last_month_users:
            tag = "留存客户"
        else:
            tag = "唤醒客户"
        res_list.append({
            "month": target_month,
            "user_id": uid,
            "customer_type": tag
        })
    return pd.DataFrame(res_list)
04 UV 深挖：渠道 × 活动类型归因
分析思路

确认新客户流失是主因后，下一个问题是：从哪个渠道来的新客户减少了？哪个具体的活动类型（campaign_type）贡献了最大的下滑？

先锁定"新客户"群体（用 inner join 把用户标签表接回行为主表），再在这个子集内做渠道维度的透视，然后再往下钻一层到 channel × campaign_type 的二维交叉。

遇到的问题 · 下钻逻辑的一致性

渠道层的分析用的是"新客户"子集。下钻到 campaign 层时，需要保持同样的子集范围，即仍然只看新客户，而不是全量用户。如果不注意这一点，campaign 层的数字就会比渠道层膨胀，两层之间无法对齐。

解决方式是：campaign 分析沿用同一张 df_all_new（已 inner join 过用户标签、仅保留新客），直接在这张表上做 pivot，不再引入新的 df 过滤逻辑。

分析结论

新客户下滑从严重到轻排序：Paid Ads → Organic → Social → Email。下钻到活动类型后，Paid Ads × Discount Campaign 是最大的单一下滑来源，推测原因包括：促销吸引力下降、预算不足、广告素材效率下降（CTR/CPC 维度可进一步验证）。

05 CVR 漏斗分析
为什么继续看 CVR

连锁替代法显示 CVR 的贡献不是主因，但并不意味着 CVR 没有问题——它可能在某些环节、某些用户群体上有显著恶化，只是被整体数字平均掉了。

漏斗分 5 个步骤：首页访问 → 详情页访问 → 加购 → 开始结算 → 完成支付。我在整体对比的基础上，进一步拆解了每个环节对总 CVR 变动的贡献，并按用户类型做了切片分析。

遇到的问题 · 漏斗各步骤的去重逻辑

漏斗数据中，每一步的标记是 True/False。由于同一用户有多条记录，不能直接 sum，否则一个用户多次到达同一步骤会被重复计入。

正确做法：先 groupby('user_id') 对各步骤列取 max()，得到"这个用户这个月是否曾经到达过该步骤"，再 sum() 算人头。每个用户在每个步骤只计一次。

遇到的问题 · 体量加权的必要性

在按用户类型切片分析环节贡献时，需要额外考虑体量加权：同样转化率下降 5 个百分点，如果新客户人数是留存客户的三倍，新客户的拖累就会三倍于留存客户。所以用"各客群在环节入口的平均体量占比"作为权重，才能得到对大盘的真实影响量。

分析结论

CVR 下滑主要集中在 2→3 环节（详情页→加购），新客户是主要拖累来源。推测方向：商品吸引力、价格竞争力、详情页体验（停留时长、跳出率）。

cvr_funnel.py · 漏斗去重+环节贡献拆解 
def get_monthly_funnel_rates(df, month_str):
    """返回某月各漏斗环节的去重转化率"""
    m_df = df[df['month'] == month_str].copy()
    cols = ['visited_website', 'viewed_product',
            'added_to_cart', 'checkout_started',
            'purchase_completed']
    m_df[cols] = m_df[cols].astype(int)

    # 关键去重：groupby → max → sum
    dedup = m_df.groupby('user_id')[cols].max()
    s1,s2,s3,s4,s5 = [dedup[c].sum() for c in cols]
    return {
        'r1':s2/s1, 'r2':s3/s2,
        'r3':s4/s3, 'r4':s5/s4,
        'overall': s5/s1}

base   = get_monthly_funnel_rates(df, '2025-11')
target = get_monthly_funnel_rates(df, '2025-12')

# 连锁替代：逐步替换各环节
c1 = (target['r1']*base['r2']*base['r3']*base['r4']) - base['overall']
c2 = (target['r1']*target['r2']*base['r3']*base['r4']) \
   - (target['r1']*base['r2']*base['r3']*base['r4'])
c3 = (target['r1']*target['r2']*target['r3']*base['r4']) \
   - (target['r1']*target['r2']*base['r3']*base['r4'])
c4 = target['overall'] \

06 综合结论与后续验证方向
根因链

GMV 下滑 → UV 流量是主因（贡献 123%） → 流量里新客户流失最严重 → 新客户主要从 Paid Ads 渠道进来 → 具体是 Discount Campaign 活动的新客获取量大幅下滑。

CVR 下滑是次要问题，集中在 详情页→加购 环节，同样以新客户为主。

后续可验证的假设

Paid Ads 侧：促销力度下降（折扣幅度、SKU 数量）、投放预算不足、广告素材效率下降——可通过 CTR、CPC、曝光量等广告投放数据验证。

CVR 侧：商品价格竞争力、详情页内容质量、页面加载速度——可通过页面停留时长、跳出率、价格对比行为数据验证。

分析边界说明

本分析基于用户行为漏斗数据，没有接入广告投放数据（曝光/点击/CTR/CPC）和页面体验数据（停留时长/跳出率），因此结论停留在"指出方向"层面，无法直接量化广告效率恶化的具体程度。上述验证方向需要补充对应数据源才能推进。
